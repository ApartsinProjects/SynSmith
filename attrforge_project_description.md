# AttrForge: Multi-Objective Prompt Debugging for Realistic, Diverse, and Attribute-Controlled Synthetic Data Generation

## 1. Project Overview

**AttrForge** is a framework for generating synthetic data with known, controllable attributes while ensuring that the generated data is both realistic and diverse.

The project treats synthetic data generation as an iterative **prompt debugging** process. Instead of writing a single generation prompt manually, the system continuously improves the prompt using feedback from several LLM-based critics:

1. **Attribute Verifier** — checks whether each synthetic sample matches the requested attributes.
2. **Realism Discriminator** — compares real and synthetic samples and detects synthetic artifacts.
3. **Diversity Auditor** — analyzes the generated dataset for missing modes, repetition, imbalance, and shallow variation.
4. **Prompt Updater** — rewrites the generator prompt using structured feedback from the critics.

The goal is to produce synthetic datasets that are:

- **attribute-controlled**;
- **realistic**;
- **diverse**;
- **useful for downstream training and evaluation**.

---

## 2. Motivation

Synthetic data is increasingly used when real data is scarce, expensive, private, sensitive, or difficult to label. However, synthetic data generated by LLMs often suffers from several limitations:

- samples are too clean or too polished;
- examples are repetitive;
- labels are too obvious;
- rare or ambiguous cases are missing;
- generated samples do not match the intended attributes;
- the data distribution differs from real data;
- diversity is superficial rather than semantic;
- synthetic artifacts are easily detected by humans or models.

A major weakness of naive synthetic data generation is that the prompt is usually created manually and remains fixed. If the generated data is unrealistic or repetitive, the improvement process depends heavily on human intuition.

AttrForge addresses this problem by turning prompt improvement into a systematic feedback loop.

---

## 3. Core Objective

The objective of this project is to develop and evaluate a framework for generating synthetic data that satisfies three goals simultaneously:

1. **Attribute fidelity**  
   Each generated sample should match a predefined attribute vector.

2. **Realism**  
   Synthetic samples should be difficult to distinguish from real examples.

3. **Diversity**  
   The generated dataset should cover a broad range of valid labels, styles, difficulty levels, ambiguity levels, scenarios, and edge cases.

The central research question is:

> Can synthetic data generation be improved by iteratively debugging the generation prompt using feedback from attribute, realism, and diversity critics?

---

## 4. Problem Definition

Given:

- a small set of real examples `R`;
- an attribute schema `A`;
- an initial generation prompt `P0`;
- a generator LLM `G`;
- a set of LLM-based critics;

the goal is to iteratively refine the prompt `P` so that the generated synthetic dataset `S` maximizes:

```text
attribute fidelity + realism + diversity + downstream usefulness
```

More formally:

```text
S = G(P, A)
```

where every generated sample has an intended attribute vector:

```json
{
  "label": "Class B",
  "difficulty": "hard",
  "ambiguity": "high",
  "style": "informal",
  "noise": "missing_details",
  "scenario_type": "edge_case"
}
```

The system must ensure that the output sample actually reflects these attributes.

```text

       +

       ↓

       ↓

       ↓
┌──────────────────────────────┐
```

## 7. Main Components

### 7.1 Attribute Planner

The attribute planner decides which attribute combinations should be generated.

Instead of asking the generator to produce “diverse examples” in an open-ended way, the planner creates explicit target vectors.

Example:

```json
{
  "sample_id": "target_017",
  "label": "negative",
  "difficulty": "hard",
  "ambiguity": "high",
  "style": "fragmented",
  "noise": "missing_details",
  "scenario_type": "rare"
}
```

This enables controlled generation and allows later evaluation of whether the generator followed the requested attributes.

---

### 7.2 Generator LLM

The generator receives:

- the task description;
- a few real examples;
- the attribute schema;
- a target attribute vector;
- the current generation prompt.

It outputs a synthetic sample and its metadata.

Example output:

```json
{
  "text": "I mean, it worked at first, but now it randomly stops and nobody explains why. Maybe I missed something, but this is getting annoying.",
  "label": "negative",
  "difficulty": "medium",
  "ambiguity": "medium",
  "style": "informal",
  "noise": "missing_details",
  "scenario_type": "common"
}
```

---

### 7.3 Attribute Verifier

The attribute verifier checks whether the generated sample actually matches the requested attributes.

Example verifier output:

```json
{
  "attribute_match": false,
  "failed_attributes": [
    "difficulty",
    "ambiguity"
  ],
  "reason": "The sample is labeled as hard and highly ambiguous, but the sentiment is directly expressed and easy to identify."
}
```

This component prevents the generator from merely attaching metadata without producing text that truly reflects the attributes.

---

### 7.4 Realism Discriminator

The realism discriminator receives a shuffled mixture of real and synthetic samples.

Its task is to classify each sample as:

```text
real
synthetic
```

For synthetic samples that are correctly detected, the discriminator provides an explanation.

Example:

```json
{
  "sample_id": "S42",
  "prediction": "synthetic",
  "confidence": 0.91,
  "reason": "The sample is too polished, too complete, and follows a predictable structure. Real examples contain more incomplete thoughts and inconsistent phrasing."
}
```

These explanations are used to identify synthetic artifacts.

Common artifacts include:

| Artifact Type | Description |
|---|---|
| Over-polished language | Text is too clean or grammatically perfect |
| Repetitive structure | Many samples follow the same template |
| Excessive clarity | The intended label is too obvious |
| Missing ambiguity | Real-world uncertainty is absent |
| Unrealistic completeness | Every example contains all expected details |
| Generic vocabulary | Domain-specific language is missing |
| Balanced artificial structure | The sample looks intentionally constructed |
| Lack of noise | No typos, omissions, contradictions, or irrelevant details |

---

### 7.5 Diversity Auditor

The diversity auditor evaluates the synthetic dataset as a whole.

It checks whether the generated batch covers the target space sufficiently.

The auditor looks for:

- missing attribute values;
- missing attribute combinations;
- overrepresented modes;
- near-duplicate samples;
- shallow paraphrases;
- semantic repetition;
- lack of rare or edge cases;
- unrealistic over-diversification.

Example diversity audit:

```json
{
  "summary": "The batch is diverse in wording but narrow in scenario type.",
  "missing_modes": [
    "high-ambiguity negative examples",
    "short fragmented examples",
    "rare scenarios with missing details"
  ],
  "overrepresented_modes": [
    "clean medium-length examples",
    "directly stated sentiment",
    "single-sentence complaints"
  ],
  "near_duplicate_rate": 0.22,
  "recommendations": [
    "Force generation across explicit scenario families.",
    "Increase the proportion of fragmented and incomplete examples.",
    "Avoid reusing complaint templates."
  ]
}
```

---

### 7.6 Prompt Updater

The prompt updater receives feedback from:

1. the attribute verifier;
2. the realism discriminator;
3. the diversity auditor.

It then rewrites the generation prompt.

The updater must preserve the original task while improving the prompt along three dimensions:

```text
1. better attribute control
2. higher realism
3. broader diversity
```

Example prompt update:

Before:

```text
Generate realistic customer complaints with the requested label and style.
```

After:

```text
Generate customer complaints that match the requested label, difficulty, ambiguity, style, noise level, and scenario type.

Do not make the label obvious unless the target difficulty is easy.
For hard or high-ambiguity examples, include mixed signals, indirect wording, missing context, or uncertainty.

Vary the structure, length, emotional tone, and scenario family.
Avoid repeating the same complaint template.
Some examples should be incomplete, informal, or contain irrelevant details, as real customer messages often do.
```

## 8. Iterative Algorithm

```text
Input:
  R  = small real dataset
  A  = attribute schema
  P0 = initial generator prompt
  G  = generator LLM
  V  = attribute verifier
  D  = realism discriminator
  C  = diversity auditor
  U  = prompt updater
  T  = number of iterations

For t = 0 ... T:

  1. Select target attribute vectors from A.

  2. Generate synthetic samples:
       S_t = G(P_t, target_attributes)

  3. Verify attribute fidelity:
       V(S_t, target_attributes)

  4. Build a mixed real/synthetic batch:
       M_t = shuffle(R_eval ∪ S_t)

  5. Run realism discrimination:
       D(M_t)

  6. Run diversity audit:
       C(S_t, A)

  7. Aggregate feedback:
       - attribute failures
       - realism artifacts
       - missing modes
       - overrepresented modes
       - duplicate patterns

  8. Update the prompt:
       P_{t+1} = U(P_t, feedback)

  9. Repeat generation with the updated prompt.

Output:
  Final prompt P_T
  Synthetic dataset S_T
  Attribute fidelity report
  Realism report
  Diversity report
  Downstream evaluation results
```

## 9. Research Questions

### RQ1: Attribute Fidelity

Can the framework generate samples that reliably match their requested attribute vectors?

Possible metrics:

- attribute match accuracy;
- per-attribute precision, recall, and F1;
- attribute violation rate;
- confusion matrix between requested and verified attributes.

---

### RQ2: Realism

Does discriminator-guided prompt debugging make synthetic samples harder to distinguish from real samples?

Possible metrics:

- real-vs-synthetic discriminator accuracy;
- discriminator confidence;
- human realism score;
- LLM realism score;
- realism improvement over iterations.

A successful realism loop should reduce discriminator accuracy toward chance level in a balanced real/synthetic setting.

---

### RQ3: Diversity

Does coverage-guided prompt debugging improve the diversity of generated data?

Possible metrics:

- attribute entropy;
- coverage of valid attribute combinations;
- near-duplicate rate;
- embedding-cluster coverage;
- semantic dispersion;
- lexical diversity;
- distribution over scenario types;
- rare-case coverage.

---

### RQ4: Downstream Usefulness

Does the improved synthetic data help downstream models?

Possible evaluation setups:

| Training Data | Test Data |
|---|---|
| Small real set only | Held-out real set |
| Naive synthetic data | Held-out real set |
| AttrForge synthetic data | Held-out real set |
| Small real + naive synthetic | Held-out real set |
| Small real + AttrForge synthetic | Held-out real set |

Possible downstream metrics:

- accuracy;
- macro-F1;
- AUROC;
- AUPRC;
- extraction F1;
- robustness on rare or ambiguous subsets.

---

## 10. Baselines

The project should compare AttrForge against several baselines:

| Baseline | Description |
|---|---|
| Naive prompting | One manually written prompt |
| Few-shot prompting | Prompt includes real examples but no iterative refinement |
| Self-critique prompting | Generator critiques and improves its own output |
| Diversity-only prompting | Prompt explicitly asks for diversity but has no audit loop |
| Realism-only prompting | Uses realism feedback but no attribute or diversity control |
| Human prompt refinement | Human manually improves the prompt |
| AttrForge | Attribute, realism, and diversity feedback loops combined |

## 11. Evaluation Plan

### Phase 1: Small Pilot

Use one narrow domain and one task.

Example:

```text
Domain: customer support tickets
Task: intent classification
Attributes: intent, difficulty, ambiguity, style, noise, scenario type
```

Evaluate whether the system can generate samples that match requested attributes.

---

### Phase 2: Realism Evaluation

Create mixed batches of real and synthetic samples.

Evaluate whether:

- the realism discriminator becomes less accurate over iterations;
- human judges find later synthetic samples more realistic;
- synthetic artifacts become less frequent.

---

### Phase 3: Diversity Evaluation

Measure whether generated data covers the intended attribute space.

Analyze:

- missing modes;
- repeated templates;
- near duplicates;
- semantic clusters;
- rare-case coverage.

---

### Phase 4: Downstream Evaluation

Train task models using different synthetic datasets and test on held-out real data.

Compare:

```text
naive synthetic vs. AttrForge synthetic
```

The main practical success criterion is improvement on real test data, especially rare, hard, or ambiguous cases.

## 13. Main Novelty

The novelty is not simply using LLMs to generate synthetic data.

The main novelty is framing synthetic data generation as:

```text
multi-objective prompt debugging
```

The prompt is optimized using feedback from multiple critics:

```text
attribute verifier + realism discriminator + diversity auditor
```

This creates a GAN-like process, but instead of updating neural network weights, the system updates the generation prompt.

The framework moves synthetic data generation from:

```text
manual prompting
```

to:

```text
iterative, feedback-driven prompt refinement
```

## 14. Risks and Challenges

| Risk | Description | Mitigation |
|---|---|---|
| LLM judge bias | The verifier or discriminator may be unreliable | Use multiple judges, repeated runs, and human validation |
| Overfitting to discriminator | Generator learns to fool the judge but not humans | Use held-out judges and human review |
| Unrealistic diversity | Prompt forces rare combinations that do not occur naturally | Define valid attribute combinations carefully |
| Attribute conflicts | Some attributes may be incompatible | Add schema constraints |
| Prompt bloat | Prompt grows too long after many updates | Use prompt compression and version control |
| Mode chasing | System overcorrects after each audit | Use aggregate feedback across batches |
| Label leakage | Generated text makes the label too obvious | Add difficulty and ambiguity controls |

## 20. Short Abstract

Synthetic data generated by large language models is often attractive when real labeled data is scarce, but naive prompting frequently produces samples that are repetitive, overly clean, unrealistic, or poorly aligned with intended attributes. This project proposes **AttrForge**, a framework for attribute-controlled synthetic data generation through multi-objective prompt debugging. Starting from a small set of real examples and a user-defined attribute schema, AttrForge generates synthetic samples with explicit target attributes. Three LLM-based critics then evaluate the generated data: an attribute verifier checks whether samples match their requested attributes, a realism discriminator identifies artifacts that distinguish synthetic from real examples, and a diversity auditor detects missing modes, imbalance, and repetition. Their structured feedback is used by a prompt updater to iteratively revise the generator prompt. The framework is evaluated through attribute fidelity, realism, diversity, and downstream task performance on held-out real data. AttrForge reframes synthetic data generation as an iterative prompt optimization process, aiming to produce synthetic datasets that are controllable, realistic, diverse, and useful for training and evaluation.
